# FE-04 Capstone Skeleton, Deployed

## 1. Assignment Overview
This assignment establishes the foundation for the Capstone project—an AI Productivity Dashboard. We migrated from a Vite-based setup to a Next.js App Router skeleton equipped with Tailwind CSS and established a responsive, route-ready framework designed for server-rendered performance and eventual AI feature integration.

## 2. Tech Stack
- **Framework:** Next.js (App Router)
- **Language:** JavaScript
- **Styling:** Tailwind CSS
- **Linting:** ESLint
- **Package Manager:** npm

## 3. Routes Created
The following placeholder routes were implemented:
- `/` - Landing Page
- `/dashboard` - Dashboard overview
- `/tasks` - Task management placeholder
- `/insights` - AI insights placeholder
- `/settings` - Settings placeholder
- `/profile` - User profile placeholder
- `/health` - Health-check page
- `/api/health` - Health API endpoint

## 4. Root Layout and Navigation
The root layout (`src/app/layout.js`) provides a consistent shell for the application:
- Includes a global `<Navbar />` with links to all core routes.
- Defines a centralized `<main>` content area.
- Includes a simple semantic `<footer>`.
- Responsive navigation tested for mobile (375px) and desktop (1280px) breakpoints.

## 5. Tailwind Design Tokens
Global design tokens are set up in `src/app/globals.css` using standard CSS variables (e.g., `--background`, `--foreground`, `--primary`, `--card`, etc.) allowing seamless transition between light and dark modes out-of-the-box, ensuring a clean, modern, and accessible design system.

## 6. Health Check Page
The `/health` page renders application status using data from a shared helper (`src/lib/health.js`). 
**Note:** Instead of using `fetch('/api/health')` inside the Server Component, the `/health` page imports the `getHealthData()` helper directly. This is a best practice in Next.js App Router, as fetching absolute URLs to internal API routes during static generation (build time) will fail if the server is not actively running.

## 7. Environment Variable Structure
Environment variables are handled securely:
- Provided an `.env.example` mapping out expected keys (`NEXT_PUBLIC_APP_NAME`, `NEXT_PUBLIC_APP_ENV`).
- Ensured `.gitignore` explicitly blocks `.env`, `.env.local`, and other secret files from being tracked.

## 8. Deployment Target
This application is optimized and intended for deployment via **Vercel**, enabling zero-configuration builds and branch previews on every push.

## 9. Responsive Verification
Structural testing confirms the layout, navigation, and placeholder cards scale appropriately from mobile viewports (375px) up to desktop resolutions (1280px).

## 10. No-Secrets Note
No secrets, private API keys, or production `.env` files are tracked or committed in this repository. 

## 11. Build Verification
The application has been verified to build cleanly using `npm run build` and conforms to ESLint standards via `npm run lint`.
